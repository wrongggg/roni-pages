What "smart" means - 3 caching strategies

1. Cache repeat adds - When the same user adds the same track again (e.g. same intro music every episode), serve from local cache. Data shows the median user adds the same single track 3 times/month. This alone saves ~50% of all add events.

2. Cache preview to add - When a user previews a track and then adds it to their timeline, the file is already fetched from the API for the preview. No second API call needed. Covers ~60% of remaining unique adds.

3. Cache repeat previews - When a user previews the same track multiple times in a browsing session. Saves ~24% of preview events.

How we calculated this

Source: Mixpanel production data, last 30 days (BlueDot music library events).

Raw events: 64K preview clicks + 48K track adds = 112K total CDN calls/month if every action hits the API.

After smart caching:

Unique user-track preview pairs: 11.6K users x 4.2 unique tracks = ~49K (vs 64K raw previews)

Unique user-track add pairs: 9.5K users x 2.4 unique tracks = ~23K (vs 48K raw adds)

Of those 23K unique adds, ~60% were on tracks already previewed by that user = served from cache.

Net unique API calls: ~49K previews + ~9K direct adds = ~58K/month

Growth assumptions

Music usage growth: 20% increase from better library quality, improved search and filtering.

SFX adoption: New category. Estimated 20% of users adopt, adding ~15-20K SFX calls/month.

Combined growth estimate: 58K current smart -> 90K/month with growth.
