What "smart" means - session-level caching only

Constraint: We cannot store Soundstripe tracks on our servers. Caching is limited to same-session only. Repeat adds across sessions (e.g. same intro every episode) require a fresh API call each time.

1. Cache preview to add (same session) - When a user previews a track and then adds it to their timeline in the same session, the file is already fetched. No second API call needed. Saves ~14K adds/month.

2. Cache repeat previews (same session) - When a user previews the same track multiple times while browsing in one session. Saves ~8K previews/month.

Total session-level savings: ~22K/month (vs ~54K with persistent caching).

How we calculated this

Source: Mixpanel production data, last 30 days (BlueDot music library events).

Raw events: 64K preview clicks + 48K track adds = 112K total CDN calls/month if every action hits the API.

After session-level caching:

Previews: 64K raw - ~8K same-session repeat previews = ~56K

Adds: 48K raw. Of those, ~14K follow a preview in the same session (already cached). Remaining ~34K adds require a fresh API call (includes all repeat adds across sessions, e.g. same intro track every episode).

Net API calls: ~56K previews + ~34K adds = ~90K/month

Note: 57% of users use just 1 track repeatedly. With persistent caching this would have saved ~25K calls/month, but the contract doesn't allow storing tracks.

Growth assumptions

Music usage growth: 20% increase from better library quality, improved search and filtering. 90K smart -> ~108K/month.

SFX adoption: New category. ~10K users estimated (conservative, based on ~20K total music users across BlueDot + in-house library today). SFX browsing is heavier on previews but fewer adds per user. ~45K CDN calls/month after session caching.

Combined growth estimate: 108K music + 45K SFX = ~150K/month (smart). Raw without caching: ~180K/month.
